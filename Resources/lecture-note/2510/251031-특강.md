# K - 애자일

1. 수직적 의사결정 구조
정말 잘나가는 신생 스타트업의 Co-Founder의 출신들 보면 대기업에서 창업하러 넘어온 사람들임

2. 실패에 관대하지 않은 문화
수직적 기업에서 오다보니 수직적으로 잡을 수 밖에 없다.
실패에 관대하지 않음.
실패에 원인을 찾고 개선하지라기 보다 실패한 사람을 인사 승진 누락시킨다던지 질책하는 문화가 많다.

3. 팀보다는 개인에 대한 평가와 보상
팀 전체에 대한 보상이 있어야 하는데 팀 평가가 없고 개인 평가만 있다.
팀 전체의 유기적으로 흘러가는 문화가 부족

4. 직무간 경계가 뚜렷, 견고
기획자, 개발자, 디자이너 등 직군이 많이 나뉘어져 있을 뿐더러 role, 업무 책임이 모호하지 않다.

5. 리더의 역량 부족
한국에서 애자일하게 해서 성공한 리더들이 별로 없다.
그래서 비싼 인력들을 데려오는 이유가 이런 것임.
한국은 해당 문하가 존재하지 않는다.
그런 사람들의 role은 이것들임. 개발을 잘해서가 아니다. 나이 많은은 사람들은 좀 리딩을 해야 하는 사람들임...

6. 올바른 회고 문화 부재
극단적으로 서로를 비방하는 경우
회고를 하는데 지속적으로 똑같은 문제점이 나오는 경우

# 좋은 개발 문화를 가진 회사?

1. 코드 퀄리티가 비슷한 조직
코드 자체가 일정해서 주니어가 이해하기 쉽고 코드 컨벤션 같은 게 잘 짜여있는 조직

2. 기술 공유가 활발한 조직
기술 세션을 통해 발표하거나, 외부 세션을 통해 이걸 공유하려는 조직은 개발 문화 자체가 좋았던 케이스가 많음.

3. DevOps의 역량이 좋은 조직
개발자가 인프라적 요소에 신경쓰지 않고 개발에 집중할 수 있기 때문. 60-70을 인프라에 쏟는 경우도 있었음. 이 경우 개발 문화를 갖추기 힘들다.

4. 스터디가 자주 있는 조직
BFF라는 패턴이라는 게 있다.
MSA 서비스 특성상 FE가 api 하나씩 끌어오면 관리 어려움. 중간에 게이트웨이 같은 것을 둬서 요청들을 aggregation해서 던져주는 것이 BFF 서버.
비동기 유리? WebFlux 쓰자! 써본 사람? ㅇㅋ 같이 스터디 하자! 이런 흐름이 잡힌다.
스터디하다보면 컨벤션도 잡히게됨.
이런 표현 위험하다. 이거 우리 컨벤션에 추가하자. 이런 것은 개발문화에 좋은 경험.

5. 문서가 단순한 조직
단순하게 핵심적인 내용만 기재되어있는 회사일 수록 문화가 좋았다.

# Design Review

1. 어떤 것에 대한 리뷰인지 목적을 분명히 할 것
	- 기능 검증 / 아키텍처 리뷰 / 운영,배포 전략 리뷰 등
2. 다이어그램은 필수적 제공
	- Component Diagram, Sequence Diagram/Class Diagram
	- 개발자 리뷰 받으려면 필수적
	- 개발자가 아닌 사람을 위해 컴포넌트 다이어그램 명시 필요
3. 설득해야 할 내용에 대한 확실한 근거 마련
	- 예상되는 Trade-Off가 무엇이 있고 대안이 무엇인지 등
	- 리뷰할 때 압박 장난 아님.
	- ex) 카프카 요청. 엥? 배치쓰면 안됨? 꼭 필요함? -> 저희 모델이 메시징 큐 써서 비동기 쓰는 거라 실시간 성 데이터 보장필요해서... 같은 근거가 설득 근거가 됨.
4. 운영이나 장애 대응 전략도 언급할 것
	- 서킷 브레이커 같은 장애 대응 전략 
5. 틀린 부분을 찾는 게 아니라 더 나은 선택을 하기 위한 과정
	- 문제를 찾는 사람들이 있는데 문제를 찾아서 더 나은 선택을 하는 과정이다~
